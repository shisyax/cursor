# Agents.md

このリポジトリはドキュメントプロジェクトです。

## 基本方針

- テストは不要です。
- PR 本文に `Tests` セクションは書きません。
- PR 本文には修正点、レビュー対応、作業履歴を書きません。
- PR の Summary には、最終的に追加または変更された成果物だけを書きます。

## 日本語文体

- 読者が普通に読める説明文を優先します。
- 指摘された語だけを別の語に置き換える作業は禁止です。
- 一文ごとに、その文が何を説明するための文かを決めてから書きます。
- 短い断定を並べません。動作、結果、作品上の意味がつながる文にします。
- 汎用語で文を終わらせません。対象が何をどう変えるのか、どの描写につながるのかまで書きます。
- 受身表現や状態説明が続く場合は、語尾だけで直さず、段落や箇条書き全体を組み直します。
- 主観的な感想は書きません。ただし、客観性を意識しすぎて報告書のような硬い文にしません。
- 抽象名詞に頼りません。文脈上確認できる描写、流れ、関係を具体的に書きます。
- 箇条書きでも、説明量と粒度を揃えます。短い項目だけが浮く場合は、前後の文脈を補います。

## 文体の判断基準

文を直すときは、特定の語句だけを潰しません。次の基準で文全体を判断し、外れる文はすべて書き直します。指摘された語と同じ「型」の文を、全文から探して直します。

### 視点

- 作品の外から眺めて解説しません。「作品が〜を描く」「物語が〜になる」「〜を強調している」のような、外側からの説明をしません。
- 作中で実際に起きること、とくにベンの身に起きることを、そのまま書きます。

### 態と主語

- 装置・物語・作品を主語に立てて、「〜を起こす」「〜を発動する」「〜を当てる」のように動かしません。
- 受身（`される`）と使役（`させる`）は使いません。ベン自身や、起きる出来事そのものを主語にします。

### 語彙

- 硬い動詞や改まった言い回し（例: `起こす`／`発動する`／`扱う`／`圧迫する`／`連想させる`／`姿を変える`）を避けます。
- 抽象的な締め（例: `軸になる`／`要素になる`／`焦点を当てる`／`手がかりになる`）を避けます。
- ふつうに声に出して読める語を選びます。

### トーン

- 報告書のように硬く、分析的にしません。
- 同時に、主観的な感想も足しません。

語句を別の語へ置き換えるだけの作業はしません。基準から外れる文は、文全体を組み直します。

## 引き継ぎ手順

1. `Agents.md` を先に読みます。
2. 指摘コメントの有無に関係なく、対象 Markdown 全体を通読します。
3. 指摘された語だけを直しません。その語が表す「型」（視点・態・語彙・トーン）を基準で捉え、同じ型の文を全文から探します。
4. 全文を一度に書き換える危険を避け、節ごとに区切って、その節の全文を基準に照らして書き直します。
5. 書き直したあと、同じ語尾、同じ文型、短い断定が続いていないか確認します。
6. 仕上げに、語句検索・全文通読・語尾統一の確認を 3 回くり返します。
7. PR 本文には修正内容を書きません。

## 最終報告

- 最終報告は、実施した成果だけを簡潔に書きます。
- 「確認のみ行いました」「追記していません」のような報告だけの文は書きません。
- 「テストは実行していません」のような実施有無だけの文は書きません。
- ユーザーが求めていない補足説明は省きます。
